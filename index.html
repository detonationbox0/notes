What Do I Know?

My focus always has been on dependency-free solutions. These have largely been confined by the tools the operating system and relevant software has at hand. To be maximally effective, without the overhead of convincing my superiors of the safety of third-party software, was to use what is already available.

MacOS has it's own automation language, AppleScript. It is also a Unix system, which means it has a Unix shell. This Unix shell has a variety of programming languages built-in such as Bash, Python , Ruby and others.

Adobe also has scripting capability built into it's applications. This API can be interfaced using pure AppleScript, JavaScript (in the form of ExtendScript), or it's own extensibile platform Adobe CEP. I have become comfortable with these.

I've also learned some frameworks that allow me to compile installable and sharable MacOS applications that can interface with the operating system, Electron I've even used Xcode , though just to make Cocoa-AppleScript applications. Though, not willing to shell out $100/yr, I've been behind the Gatekeeper, needing my users to do something in System Preferences to get my applications to run.

Oh, and this document you are reading? I've become a huge lover of Markdown.

Marco's Creator

This application is used by the Account Specialists to create artwork on-the-fly for Marco's products which contain sufficiently predictable content changes.

There have been many versions of this application, it has a storied history.

Each Account Specialist maintains a table of their clients, which includes information like contact information, coupons, offer price points, even menu content. They may change any of this information, and request an artwork proof at any time. A few seconds later, a print-ready PDF of their artwork appears in their own folder synced through SharePoint.

It was created using the Electron framework. It uses SQLite to manage the databases. It is currently using OneDrive to upload the artwork requests, and download the resulting artwork. It uses a Bash script to recieve the requests. It uses Python to manage and sort the requests into a job queue. And AppleScript to execute the requests and tell InDesign what to do.

This application used to work better when each Account Specialist worked from the same database file (using a single SQLite .db file and a common server). However, in a rush to accomodate work-from-home because of the pandemic, it has come to rely on OneDrive and Sharepoint for requests and artwork retrieval - this is less efficient.

Metrix Day Extension

This is a utility used by the Prepress Artists who are responsible for "flattening" each week's artwork that is sent to the presses.

This task used to be frustratingly manual, time consuming, and repetative. It would take two artists most of a Friday to complete. You can ask Anthony about the bad old days of flattening artwork. The first solution I created involved pure AppleScript and was a very clunky experience, but it did shrink a day long process into a few hours.

A few years later, I rewrote the entire program into an InDesign Extension (Adobe CEP). This InDesign Extension has the advantage of a user interface ( HTML/CSS), Node.js support, and it can execute JavaScript within InDesign using input from the interface. This allows me to do all sorts of tricks, such as DataMatrix barcode creation and placement, dynamic template creation, custom file naming, meun fold correction, etc.

Scrutinizer

Emails from Account Specialists with the subject "Is this image high resolution?" were way too frequent.

I wrote this AppleScript droplet to solve this problem. If you drag one or more images onto it, it looks at the pixel width, and pixel height, and determines it's effective PPI and it's width and height in inches at 100% scale if placed into an InDesign document.

Here was the logic:

xxxxxxxxxx
 Images with 300dpi: Excellent
 Images with < 300dpi and effective print size is < 2 inches: Terrible  
 Images with < 300dpi and effective print size is ≤ 3  and ≥ 2 inches: Poor  
 Images with < 300dpi and effective print size is ≤ 5  and ≥ 3 inches: Not Great  
 Images with < 300dpi and effective print size is ≤ 8  and ≥ 5 inches: Good 
 Images with < 300dpi and effective print size is ≤ 10  and ≥ 8 inches: Very Good
 Images with < 300dpi and effective print size is > 10 inches: Excellent

If the image is 10,000px wide, and 10,000px tall but the PPI is 72, the PPI is not a reliable indicator of the image's quality to a human's eyeball when that image is scaled down to a fraction of it's size. The logic determines an image's likely quality based on the width and height in inches when the PPI is less than 300.

I then assigned faces to the various outputs and used friendly messaging to make the app less abbraisive and more fun.
